# Ответы на вопросы и возражения

Дата обновления: 2026-07-22

Этот раздел содержит партнёрски безопасные модели ответа на типовые вопросы
клиентов. Это не готовые обещания: формулировку нужно адаптировать к ситуации, а
условия по данным, инфраструктуре, LLM, SLA и внедрению — сверять с
[ограничениями и условиями](../legal/claims-and-limitations.md).

## Как использовать

- Начинайте с короткого ответа, затем раскрывайте только интересующие клиента
  детали.
- Отделяйте подтверждённые возможности Искры от условий конкретного проекта.
- Не называйте возможность уникальной для всего рынка без актуального
  сравнительного исследования.
- Не обещайте, что сложная интеграция будет выполнена без обследования или
  участия технической команды.

## «Все корпоративные AI-платформы похожи. Чем Искра отличается?»

### Короткий ответ

В Искре есть необходимый гигиенический минимум корпоративной AI-платформы:
AI-чат, многошаговый агент, работа с документами и знаниями, организационные
роли, навыки, API-интеграции и обсуждаемые варианты развёртывания.

Первое ключевое отличие Искры — создание новых исполняемых навыков
непосредственно из рабочего диалога. Пользователь описывает задачу обычным
языком, а ассистент может сформировать программную логику, собрать и проверить
навык. Навык получает версию, настройки и правила доступа; его можно оставить
личным или опубликовать для организации.

Это позволяет автоматизировать длинный хвост небольших и меняющихся рабочих
задач, для которых разработка отдельного жёсткого процесса может быть слишком
долгой и дорогой.

Второе отличие — независимость рабочих сценариев от конкретной нейросети или
поставщика. Платформенный администратор может менять модели и провайдеров для
настроенных LLM-маршрутов через административный интерфейс без изменения
backend-кода и перезапуска платформы. Отраслевые модели и другие нейросетевые
сервисы можно подключать как инструменты навыка через доступный API или
отдельную интеграцию, не выпуская для этого новую версию ядра Искры.

Третье отличие — Искра сама работает как агентный оркестратор. Агент может
разложить задачу на шаги, использовать доступные модели, навыки и корпоративные
инструменты в нужной последовательности, запросить подтверждение пользователя
и собрать единый результат. Для сложных инженерных задач Искра может применять
поэтапный процесс с субагентами там, где он включён.

### Расширенный ответ

Наличие агента, базы знаний, интеграций, API, ролей и разных контуров
развёртывания само по себе не является достаточным отличием. Искра объединяет
эти возможности в рабочей среде сотрудника, а первый дифференциатор —
превращение полезного рабочего сценария в повторно используемый навык прямо из
чата.

Из диалога можно создавать не только текстовые инструкции, но и программные
навыки для вычислений, обращений к внешним сервисам и многошаговых действий.
Ассистент формирует код, запускает автоматическую проверку и сборку и сообщает
пользователю готовую версию. Существующий программный навык также можно
обновить из чата.

Если навыку нужны ключи доступа, Искра предлагает защищённую форму: значения
передаются в хранилище, не показываются модели и не сохраняются в переписке.
Личный навык остаётся в профиле пользователя, а владелец или администратор
организации может создать организационный навык, обновлять его новой версией и
отзывать.

Такой подход особенно полезен для процессов, которые возникают у отдельных
сотрудников и подразделений, часто меняются и не оправдывают отдельный цикл
аналитики, разработки и внедрения для каждого сценария.

Второй дифференциатор — эксплуатационная гибкость. Для настроенных
LLM-маршрутов платформенный администратор может через интерфейс сменить backend,
URL, модель, провайдера и ключ; изменение применяется без перезапуска платформы.
Другие нейросетевые сервисы — например, отраслевую модель, распознавание
документов или речи — можно подключить через навык, инструмент или API, если у
сервиса есть совместимый интерфейс либо для него выполнена интеграция. Логика
рабочего процесса при этом остаётся в навыке и не требует изменения ядра
платформы.

Третий дифференциатор — агентная оркестрация. Искра не только передаёт запрос
одной модели: агент выполняет многошаговую задачу, выбирает из разрешённого
набора инструменты и навыки, обращается к файлам и знаниям, запрашивает
недостающие данные и собирает итоговый артефакт. Для сложных инженерных задач
доступен поэтапный процесс с планированием, независимым ревью, реализацией и
проверкой отдельными субагентами там, где включён соответствующий системный
навык.

### Гигиенический минимум Искры

- AI-чат и многошаговый агентный режим;
- работа с файлами, документами и корпоративными знаниями;
- проекты, организации, роли, группы и настройки доступа;
- навыки, API и интеграционный контур;
- cloud, российский контур и on-premise по отдельному согласованию;
- настраиваемые инструменты безопасности и Инспектор.

### Что не следует обещать

- что ни у одной другой платформы нет похожих возможностей;
- что любой сотрудник автоматизирует любой процесс без технической помощи;
- что к Искре можно подключить произвольную нейросеть без совместимого API или
  интеграции;
- что агент и субагенты могут использовать системы и данные вне выданных прав
  или без настройки соответствующих инструментов;
- что сложное подключение к внутренней системе не требует обследования;
- что выбранный контур автоматически соответствует всем требованиям ИБ или
  регулирования;
- что наличие функции само по себе гарантирует экономический эффект.

## «Каких прикладных агентов можно предоставить через API?»

### Короткий ответ

В Headless-режиме Искру можно встроить в портал, внутренний сервис, CRM, бота
или другой интерфейс клиента. Внешняя система передаёт задачу, контекст и при
необходимости файлы, а получает результат работы агента: ответ, заключение,
подготовленный материал, файл или запрос на подтверждение действия.

Агент в таком проекте — не отдельный жёстко запрограммированный бот, а
управляемый профиль Искры. Для него задаются инструкции, источники данных,
доступные инструменты, права, формат результата и правила подтверждения
действий. Типовые профили можно собирать для проверки документов, контроля
качества данных, аналитики, контроля поручений и подготовки служебных
материалов.

Headless API и прикладные профили включаются и настраиваются в рамках отдельного
интеграционного проекта.

### Расширенный ответ

Единый агентный контур Искры можно использовать для разных прикладных задач,
меняя не платформу, а профиль агента: его назначение, инструкции, знания,
инструменты, права и формат результата. Это позволяет добавлять новые сценарии
без создания отдельной агентной платформы для каждого процесса.

В качестве ориентиров для обследования можно использовать следующие профили.

#### Агент проверки комплектов документов

Проверяет комплектность, реквизиты, подписи, сроки действия, форматы, логические
противоречия и наличие обязательных сведений по заданным правилам. Возвращает
структурированное заключение с замечаниями, критичностью и основаниями. Если
результат может повлечь юридически значимое решение, окончательная проверка
остаётся за уполномоченным сотрудником.

#### Агент контроля качества реестровых данных

Находит дубли, пропуски, противоречия и некорректные значения, сопоставляет
сведения из разрешённых источников, нормализует справочные данные и предлагает
варианты исправлений. По умолчанию профиль следует проектировать для чтения;
изменения в исходной системе допускаются только в пределах выданных прав и
согласованной политики подтверждений.

#### Агент анализа данных и событий

Работает с разрешёнными базами, витринами, API и потоками событий: рассчитывает
показатели, сравнивает периоды и категории, выявляет аномалии и готовит
аналитические материалы. Для проверяемости результат должен сопровождаться
исходными данными, логикой расчёта и ссылками на источники там, где это
поддерживает интеграционный контур.

#### Агент контроля поручений

Обрабатывает протоколы и транскрипции совещаний, служебную переписку и данные
системы управления задачами. Выделяет решения, ответственных и сроки, сверяет
договорённости с существующими задачами, находит пропущенные или просроченные
поручения и готовит проекты задач и сводки по исполнению. Создание и изменение
задач выполняется только после подтверждения пользователя или по заранее
согласованной политике.

#### Агент подготовки проверяемых служебных материалов

Собирает сведения из разрешённых источников и готовит справки, аналитические
записки, проекты ответов, отчёты, протоколы и краткие материалы для
руководителя. Для существенных утверждений сохраняется связь с использованными
источниками, если источники и такой формат трассировки доступны агенту.

### Как строится работа через API

1. Информационная система передаёт задачу, контекст, параметры и при
   необходимости файлы.
2. Искра запускает доступный профиль агента и разрешённые ему инструменты.
3. Агент выполняет задачу в пределах прав API-ключа и подключённых систем.
4. Внешняя система получает результат и доступные материалы.
5. Значимое действие или запись в систему выполняется после подтверждения либо
   по явно согласованной политике.

Headless API позволяет обращаться к агенту Искры через REST API. Там, где режим
включён администратором, может использоваться OpenAI-совместимая поверхность
Chat Completions. Это не полный клон API OpenAI: состав методов, полей, работа с
файлами и артефактами, лимиты, срок жизни разговоров, журналирование и поддержка
фиксируются для конкретного проекта.

### Что важно согласовать до обещания клиенту

- какие профили уже можно собрать на доступных инструментах, а какие потребуют
  разработки или интеграции;
- какие системы, данные и файлы разрешено передавать агенту;
- какие операции доступны только для чтения, а какие допускают запись;
- где требуется подтверждение сотрудника;
- какой контур развёртывания, LLM и gateway применяются;
- какие нужны лимиты, журналирование, SLA и сопровождение;
- какие требования законодательства и информационной безопасности применимы к
  выбранному сценарию.

### Что не следует обещать

- что все перечисленные профили являются готовыми коробочными агентами;
- что Headless API автоматически включён в любом контуре Искры;
- что агент может получать доступ ко всем системам клиента без настройки прав;
- что действия записи безопасно выполнять без подтверждения или согласованной
  политики;
- что выбранная архитектура автоматически соответствует всем требованиям ИБ и
  регулирования без обследования.
