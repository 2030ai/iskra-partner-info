# [2026-06-16 12:49] Атомарные событийные документы

## Статус

Принято

## Контекст

`development-history.md` с правилом последних 10 записей и ручная нумерация ADR создавали общие точки редактирования. При параллельной работе нескольких агентов это повышает риск merge-конфликтов и требует ротации/архивации.

## Решение

Событийные документы вести как отдельные файлы с именем `YYYY-MM-DD-HHMM-short-title.md`.

Применить это к:

- `agent_docs/development-history/`
- `agent_docs/adr/`
- отчётам, аудитам и investigation logs, если такие коллекции появятся

Не задавать фиксированное количество записей для обязательного чтения. Агент сам определяет глубину просмотра по риску задачи, затронутым файлам, связанным решениям и поиску по ключевым словам.

## Обоснование

Отдельные файлы уменьшают конфликты, убирают ручную ротацию и не требуют синхронизации номеров между агентами. Timestamp сохраняет порядок без центрального индекса.

## Последствия

Положительные:

- меньше merge-конфликтов в журналах;
- не нужна архивация старых записей;
- проще создавать записи из параллельных worktree;
- история лучше ищется по имени файла, дате и содержимому.

Отрицательные:

- файлов станет больше;
- без дисциплины названий и шаблонов записи могут стать разношёрстными;
- при длинной истории агенту нужно осознанно выбирать глубину просмотра, а не полагаться на фиксированный хвост.

## Альтернативы рассмотрены

- Оставить один `development-history.md` с последними 10 записями: проще визуально, но сохраняет конфликты и требует архивации.
- Оставить ручную ADR-нумерацию: привычно, но два агента могут выбрать один и тот же следующий номер.
